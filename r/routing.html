Because it makes little to no sense

Read this before you start

This document might be unstructured and chaotic and does not include everything you need to know and may not be correct, but will probably get you back on track when you're stuck. Keep an eye out for notes like this one because it may just contain the information you need to continue with your work. You might need to also use your own ip addresses to configure certain things, so be mindful of that. Also note that there might be some errors in this document.

You might need to reset configuration if you've done something wrong. If this is the case, add no in front of the command to remove it from the configuration. For example if you've set the wrong ip address using: ip address 192.168.10.1 255.255.255.0, you can undo your config by writing a new command: no ip address 192.168.10.1 255.255.255.0.

RIPv2

Hardware layout

These are the routers and switches in this network.

Set unmatched route (default route)

The unmatched (or default) route is the route that will be taken if there's no match within the network. In this case it's at router 1:

xxxxxxxxxx
R1(config)#ip route 0.0.0.0 0.0.0.0 G0/2
%Default route without gateway, if not a point-to-point interface, may impact performance
xxxxxxxxxx
ip route 0.0.0.0 0.0.0.0 <outbound interface>

Note

The default route should only be set at the router at the edge of the network. In this case it's R1 because it's connected to the rest of the internet.

Configure RIP

This example will look at R1, but the configuration for the other routers are similar (view notes below).

xxxxxxxxxx
R1(config)#router rip
R1(config-router)#version 2
R1(config-router)#no auto-summary
R1(config-router)#network 192.168.1.0
R1(config-router)#network 192.168.2.0
R1(config-router)#passive-interface G0/1
R1(config-router)#default-information originate
xxxxxxxxxx
router rip
version 2
no auto-summary
network <connected networks...>
passive-interface <passive interfaces>
default-information originate
Note

Some important things to consider when configuring RIP:

  • network <connected network>: The connected networks are all the networks that are directly connected to this router. So in the same of R1, it's the network between S1 and the network between R2.
  • passive-interface <passive interface>: The passive interfaces should be the interface that should not be included in the RIP process. In the case of R1, it's G0/1 because that's the interface connected to the switch. If it were a router, the interface would not be set to passive.
  • default-information originate: This is only for the router that's the endpoint of the network and connects to a WAN (internet).

You can use no to remove configuration. So if I mistakenly configured network 192.168.11.0 even though it's not directly connected, type: no network 192.168.11.0. Usually just putting no in front of the command you've run by accident will revert the changes of that command.

Also remember to configure the interfaces with an ip-address and changing their state to up.

Working with underlying VLAN's

Remember to add the underlying VLAN's to the RIP configuration. So if we have the VLAN's shown in InterVLAN, remember to go into the router rip configuration and add the networks:

xxxxxxxxxx
R1(config)#router rip
R1(config-router)#network 192.168.10.0
R1(config-router)#network 192.168.20.0
R1(config-router)#network 192.168.30.0

Not doing this will prevent devices from talking to devices within the VLAN's and this includes DHCP servers. So when working with a DHCP server and a relay, remember to add it to the routers config.

Configure relay

Configure the relay by adding a helper ip-address to the inner interface. So in the following scenario where R1 is the DHCP server and R2 is an underlying router, the G0/1 interface of R2 should be configured. The IP address should be the address of the interface of R1 for that section of the network. So if R2 also had another router connected it should also use the IP address of R1. So in this case it would be the address of the G0/0 interface of R1 (ip address 192.168.2.1, check RIPv2)

xxxxxxxxxx
R2(config)#int g0/1
R2(config-subif)#ip helper-address 192.168.2.1
R2(config-subif)#exit
xxxxxxxxxx
int <inner interface>
ip helper-address <helper ip-address>
Note when working with VLAN's

Adding the helper address to the interface itself won't cut it. You'll need to add it to the sub-interfaces as well:

xxxxxxxxxx
R3(config)#int g0/2.10
R3(config-subif)#ip helper-address 192.168.2.1
R3(config-subif)#exit
R3(config)#int g0/2.20
R3(config-subif)#ip helper-address 192.168.2.1
R3(config-subif)#exit
R3(config)#int g0/2.30
R3(config-subif)#ip helper-address 192.168.2.1
R3(config-subif)#exit
R3(config)#int g0/2.88
R3(config-subif)#ip helper-address 192.168.2.1
R3(config-subif)#exit

You only need to do this of course if the DHCP server is not the trunk router. Otherwise you don't have to do this.

ACL

ACL allows you to black- or whitelist devices and networks to better security and control.

Creating a access-list

Access lists are the rules that are applied when filtering. Keep in mind that ACL will look at the source IP-address so keep that in mind when creating your access list.

When an IP-address needs to be matches, it stops the execution at the first rule it finds that applies to it, so for example if we have the following access list and there's a packet coming from `192.168.20.10:

xxxxxxxxxx
R3#show access-list 1
Standard IP access list 1
    permit 192.168.10.0 0.0.0.255 (9 match(es))
    permit 192.168.20.0 0.0.0.255 (2 match(es))
    deny any (2 match(es))

Steps

  • Checks the first rule. In this case, the rule doesn't apply to 192.168.20.10
  • Checks the second rule. This one matches the source IP-address so it stops checking the next rules.
  • The last rule is ignored since the IP-address has already been matched.

The order of the rules matter because of this, so something like:

xxxxxxxxxx
R3#show access-list 1
Standard IP access list 1
    deny any (2 match(es))
    permit 192.168.10.0 0.0.0.255 (9 match(es))
    permit 192.168.20.0 0.0.0.255 (2 match(es))

Won't work, because any address will be matched and will deny every address.

  • To permit or deny a host: access-list <#> <permit | deny> host <ip address> or access-list <#> <permit | deny> \<IP-address> 0.0.0.0

  • To deny a network: access-list <#> <permit | deny> <network address> <ignore mask>

    • The ignore mask is most of the time the inverse of the subnet mask for that network, so if the subnet mask is 255.255.255.0 the ignore mask is 0.0.0.255
xxxxxxxxxx
R2(config)#access-list 1 permit host 192.168.10.10
R2(config)#access-list 1 permit 192.168.20.0 0.0.0.255
xxxxxxxxxx
access-list <list number> permit host <IP-address>
access-list <list number> permit <network address> <ignore mask>

Applying an access-list

You need to make sure you apply the access list the right way around. You can filter the incoming traffic or the outgoing traffic. If you set it to filter the outgoing traffic for an interface, every packet that is send to the next device from that interface will be filtered before it's send. If you set it to filter incoming traffic, it'll filter all traffic send to that interface.

So if we have the following situation:

Say we want to allow both PC 2 and PC 3 to talk to PC 1, but don't want PC 2 to talk PC 3 and vice versa. We also want to block PC 1 from talking to PC 3.

xxxxxxxxxx
R2(config)#int g0/0
R2(config-if)#ip access-group 1 out
xxxxxxxxxx
int <interface>
ip access-group <list number> <in | out>

Dynamic NAT

Standard ACL (<nat list>)

First of all we need to create a pool of networks that are allowed to use NAT to talk to the internet.

xxxxxxxxxx
R2(config)#ip access-list standard R1NAT
R2(config-std-nacl)#permit 192.168.10.0 0.0.0.255
R2(config-std-nacl)#permit 192.168.20.0 0.0.0.255
R2(config-std-nacl)#permit 192.168.30.0 0.0.0.255
xxxxxxxxxx
ip access-list standard <list name>
permit <network address> <ignore mask>
Note

Permitting a single host is done using the following permit <host address> 0.0.0.0

Note

The ignore mask is usually the inverse of the subnet mask e.g. 255.255.255.0 would be 0.0.0.255

Note

The networks that should be added to the ACL should be the networks where the requests originate, so do not use the network between the switch and the router because this will not work.

Create a a IP pool (<ip pool>)

The IP pool will define the IP-addresses that will be used as the outside IP-addresses.

xxxxxxxxxx
R2(config)#ip nat pool R2POOL 202.168.10.1 202.168.10.10 netmask 255.255.255.0
xxxxxxxxxx
ip nat pool R2POOL <lower ip address> <upper ip address> netmask <subnet mask>

Adding the ACL pool to the NAT pool

xxxxxxxxxx
R2(config)#ip nat inside source list R2NAT pool R2POOL overload
xxxxxxxxxx
